# Уровни изоляции в Postgres

## 1. Что это такое

Уровень изоляции — это **контракт БД с тобой**: «если ты запустишь две транзакции параллельно, я гарантирую, что не произойдут такие-то аномалии». Чем строже уровень, тем меньше аномалий разрешено, тем дороже это стоит производительностью (больше блокировок / больше retry).

Это **не магия и не серебряная пуля**. Изоляция защищает от конкретного списка аномалий — всё, что вне списка, ты обязан защитить сам (через `FOR UPDATE`, unique constraint, оптимистичный locking, advisory locks).

## 2. Четыре аномалии

**Dirty read.** Транзакция A пишет, не коммитит. Транзакция B читает её незакоммиченные данные. Потом A откатывается — B действует на основе несуществующих данных.

**Non-repeatable read.** Транзакция A читает строку. Транзакция B меняет ту же строку и коммитит. Транзакция A читает строку снова — видит другое значение. Один и тот же `SELECT` в одной транзакции вернул разное.

**Phantom read.** Транзакция A делает `SELECT WHERE status='pending'` → 5 строк. Транзакция B вставляет новую `pending` и коммитит. Транзакция A повторяет запрос → 6 строк. Появились «фантомы» — строки, которых не было.

**Lost update.** Две транзакции читают одно и то же значение, обе изменяют его на основе прочитанного, обе пишут. Одна перезатирает другую. Именно это происходит в нашем `transferNaive`.

И есть пятая, отдельная — **write skew**. Каждая транзакция в отдельности корректна, но их комбинация нарушает инвариант. Пример: два доктора в смене, оба видят «нас двое», оба берут отгул, в итоге смена пустая. Никакой записи не потеряно — но инвариант «хотя бы один доктор на смене» сломан.

## 3. Четыре уровня по SQL стандарту

| Уровень            | Dirty | Non-repeatable | Phantom         | Lost update     | Write skew |
|--------------------|-------|----------------|-----------------|-----------------|------------|
| READ UNCOMMITTED   | ❌    | ❌             | ❌              | ❌              | ❌         |
| READ COMMITTED     | ✅    | ❌             | ❌              | ❌              | ❌         |
| REPEATABLE READ    | ✅    | ✅             | ❌ (по стандарту) | ✅ (по стандарту) | ❌         |
| SERIALIZABLE       | ✅    | ✅             | ✅              | ✅              | ✅         |

(✅ = аномалия запрещена, ❌ = разрешена)

## 4. Что реально делает Postgres

Postgres использует **MVCC** (multi-version concurrency control) — у него три уровня вместо четырёх, и они строже стандарта:

**READ UNCOMMITTED в Postgres = READ COMMITTED.** Грязного чтения в Postgres не бывает вообще, MVCC не позволяет читать незакоммиченные версии. Просто алиас.

**READ COMMITTED (дефолт).** Каждый `SELECT` видит снимок на момент **своего старта**, не на момент старта транзакции. То есть в одной транзакции два `SELECT` могут увидеть разное. Гарантия: ты не увидишь незакоммиченное.

**REPEATABLE READ.** Snapshot на момент **старта транзакции**. В одной транзакции все `SELECT` видят одно и то же. Postgres здесь строже стандарта — phantom-чтений тоже не бывает (snapshot фиксированный). Но: если две транзакции пытаются обновить одну строку, вторая получит ошибку `could not serialize access due to concurrent update` — её надо ретраить.

**SERIALIZABLE.** SSI — Serializable Snapshot Isolation. Postgres отслеживает зависимости между транзакциями и аккуратно убивает те, которые могли бы нарушить сериализуемость. Гарантия: результат идентичен какому-то последовательному выполнению. Цена: ещё больше `serialization_failure` ошибок, которые надо ретраить.

## 5. Что произойдёт в нашем сценарии на каждом уровне

Наш `transferNaive`: `BEGIN → SELECT balance → UPDATE balance = новое → COMMIT`. Две таких транзакции стартуют одновременно, обе читают `10000`, обе пишут `9999`. Потерян один перевод.

**READ COMMITTED.** Lost update **происходит**. Оба `SELECT` видят 10000, оба `UPDATE` проходят, второй перезатирает первый. БД не виновата — она дала гарантии, которые обещала.

**REPEATABLE READ.** Lost update **ловится автоматически**. Когда вторая транзакция доходит до `UPDATE`, Postgres видит, что строка уже изменена другой транзакцией с момента snapshot'а — выкидывает `40001 serialization_failure`. Ты обязан её перехватить и ретраить транзакцию целиком. Если не ретраишь — теряешь деньги пользователя по-другому.

**SERIALIZABLE.** То же самое, но детектор шире — ловит и write skew. Для двух UPDATE на одной строке поведение неотличимо от REPEATABLE READ.

## 6. Альтернативы изоляции (часто лучше)

Изоляция — не единственный способ. Для перевода между двумя счетами обычно используют:

- **`SELECT ... FOR UPDATE`** — пессимистичный замок на конкретные строки. Работает на READ COMMITTED. Никаких retry. Цена: блокировка, очереди при высокой конкуренции.
- **Атомарный UPDATE** — `UPDATE accounts SET balance = balance - $1 WHERE id = $2 AND balance >= $1`. Делает всё в одной SQL-команде, race не существует. Самое простое и быстрое. Цена: теряешь возможность сделать что-то ещё между чтением и записью.
- **Оптимистический locking через version** — `UPDATE ... WHERE version = $старая`. Если строка изменилась — обновлено 0 строк, ретраишь. Хорош под маленькую конкуренцию.
- **Advisory locks** — `pg_advisory_xact_lock(account_id)`. Когда нужен mutex по логическому ключу, а не по строке.

## 7. Главное правило

> Изоляция — это **detector**, а не **fixer**.

REPEATABLE READ не «починит» твою наивную реализацию волшебством. Он просто превратит lost update в `serialization_failure`. Если у тебя нет retry-цикла — изменения всё равно теряются, просто иначе.

Поэтому реальный продакшен-код обычно либо:
1. Атомарный UPDATE (когда возможно) — нет ни блокировок, ни retry.
2. `FOR UPDATE` + READ COMMITTED — когда нужна сложная логика между чтением и записью.
3. SERIALIZABLE + retry-цикл — когда инвариант сложный, и проще довериться БД.
